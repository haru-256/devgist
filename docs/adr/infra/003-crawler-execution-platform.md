# INFRA-ADR-003 Crawler実行基盤として Cloud Run Jobs を採用する

## Conclusion (結論)

- クローラー実行基盤として **Cloud Run Jobs** を採用する。
- 初期フェーズでは `Cloud Scheduler` + `Cloud Run Jobs` + `Artifact Registry` を基本構成とする。
- 大規模分散実行、VM レベルの制御、複雑なジョブ依存が必要になった場合は `Cloud Batch` を再検討する。

## Status

Accepted

## Context

### 背景

DevGist の `workers/crawler` は、論文・ブログなどの外部ソースを定期的に巡回し、Raw Data を収集してデータ基盤へ保存するバッチワークロードである。
このワークロードはユーザーリクエストに同期して応答する API ではなく、定期実行または手動実行されるコンテナ化バッチとして扱うのが自然である。

既存の ADR では、App Project に Crawler Jobs を配置する方針までは決定しているが、実行基盤として `Cloud Run Jobs`、`Cloud Batch`、`Cloud Run Service` のどれを採用するかは未決定だった。
本 ADR では、初期フェーズの crawler 実行基盤を決定する。

### 要件と制約

1. **定期実行との親和性**
   - 日次・週次・手動再実行を簡単に扱えること
   - 将来的に `Cloud Scheduler` から起動しやすいこと

2. **コンテナ中心の運用**
   - `workers/crawler` を Docker イメージとしてビルドし、`Artifact Registry` 経由で配布できること
   - アプリケーションコードとインフラの責務分離がしやすいこと

3. **運用負荷の低さ**
   - 個人開発の初期フェーズとして、VM 管理や複雑なジョブ編成を避けたい
   - 監視、ログ確認、再実行のオペレーションが単純であること

4. **実行時間と並列度の柔軟性**
   - crawler は I/O 主体であり、数分から数時間の実行を許容したい
   - 長時間実行が必要になっても、初期段階では十分な上限を持つこと
   - クロール対象の増加に応じて task 数や shard 数を調整できること

5. **将来の拡張性**
   - 失敗時の再実行、状態管理、ジョブ分割を段階的に導入できること
   - 将来的により大規模・高制御な基盤へ移行可能であること

### 比較した選択肢

| 選択肢 | 向いている用途 | メリット | デメリット | 今回の評価 |
|---|---|---|---|---|
| Cloud Run Jobs | コンテナ化されたバッチ、定期ジョブ、手動実行ジョブ | シンプルに始められる。コンテナ前提で開発しやすい。運用負荷が低い。task 単位で並列化しやすい。 | VM レベルの細かい制御は弱い。大規模分散や複雑なジョブ依存には向かない。 | 採用 |
| Cloud Batch | 大規模並列バッチ、HPC 寄り、VM 制御が必要なジョブ | マシンタイプ、実行環境、timeout などを細かく制御しやすい。大規模実行に強い。 | 初期構築と運用が重い。現時点の crawler には過剰設計になりやすい。 | 非採用 |
| Cloud Run Service | HTTP リクエスト駆動のアプリケーション | 既存の Web/API 実行基盤としては扱いやすい。 | バッチ実行基盤としては責務がずれる。HTTP 起動前提になり、ジョブ実行モデルと合わない。 | 非採用 |

### 選定観点

- crawler は「HPC ワークロード」ではなく「アプリケーションバッチ」であること
- コンテナとしてビルドし、定期実行・手動実行・再実行を容易にしたいこと
- 初期段階では、インフラ複雑性よりも実装と検証の速さを優先すること
- 将来的に規模や要件が増した場合は、より低レイヤな実行基盤へ移行できること

## Considered Options

### Option A: Cloud Run Jobs を採用する [採用]

`workers/crawler` をコンテナ化し、`Cloud Run Jobs` として実行する。
起動は `Cloud Scheduler` または手動実行を基本とし、イメージは `Artifact Registry` に配置する。

採用理由:

- **crawler の性質に合っている**
  - crawler は GPU を必要としない HTTP I/O 主体のバッチであり、Cloud Run Jobs のユースケースに自然に一致する。
  - コンテナ単位で完結するため、アプリケーション実装とデプロイの境界が明確になる。

- **実行時間の制約が現実的**
  - Cloud Run Jobs の task timeout はデフォルト 10 分、最大 168 時間（7日）まで設定できる。
  - 一般的な crawler バッチとしては十分に長く、「実行時間が短すぎるため候補外」とする理由はない。

- **運用が軽い**
  - VM やノードを直接管理せずに済み、初期フェーズの運用負荷を抑えられる。
  - ログ、実行履歴、再実行、引数変更などをマネージドに扱いやすい。

- **実行時オーバーライドと相性が良い**
  - task 数、引数、環境変数、timeout を execution ごとに調整できるため、クロール対象や並列度を変えながら運用しやすい。
  - shard 単位で task を分割する設計とも整合する。

- **将来の移行余地を残せる**
  - 初期は Cloud Run Jobs で素早く立ち上げ、並列度やネットワーク制御の要件が強くなった時点で Cloud Batch へ移行する判断ができる。
  - まずはシンプルな基盤で実績を作る方が、要件の実測に基づく意思決定につながる。

### Option B: Cloud Batch を採用する [却下]

`workers/crawler` を `Cloud Batch` 上で実行し、VM ベースのジョブとして管理する。

却下理由:

- **初期フェーズには制御が過剰**
  - Cloud Batch は VM レベルの制御、大規模分散実行、複雑なジョブ編成に強いが、現時点の crawler 要件にはオーバースペックである。
  - 個人開発の初期段階で必要になる運用知識と設定量が増え、ROI が低い。

- **「長時間実行のために必要」という前提が弱い**
  - Cloud Run Jobs でも task 単位で最大 7 日まで実行できるため、単に長時間実行を理由に Cloud Batch を選ぶ必要は薄い。

- **運用負荷が高い**
  - マシンタイプ、ネットワーク、ジョブ構成などの自由度が高い一方で、設計・保守の責任範囲も広がる。
  - 現時点では、crawler の本質的な課題は実行基盤よりも、再開性・冪等性・レート制御にある。

### Option C: Cloud Run Service を採用する [却下]

`workers/crawler` を常時起動の `Cloud Run Service` として配置し、HTTP リクエストまたは内部トリガーで起動する。

却下理由:

- **責務がずれる**
  - Cloud Run Service は HTTP リクエスト駆動のアプリケーション実行基盤としては適しているが、定期バッチの実行モデルとは一致しない。
  - crawler は「リクエストを捌くサービス」ではなく「完了まで走るジョブ」として扱う方が自然である。

- **運用モデルが不自然になる**
  - Scheduler から HTTP エンドポイントを叩く構成は可能だが、ジョブ実行履歴、task 並列、完了単位の管理という観点では Cloud Run Jobs の方が適している。

## Decision

DevGist の `workers/crawler` の実行基盤として、**Cloud Run Jobs を採用する**。

### 採用方針

- コンテナイメージは `Artifact Registry` に格納する
- 定期実行が必要なジョブは `Cloud Scheduler` から起動する
- 並列化が必要な場合は、Cloud Run Jobs の task を用いて shard 単位に処理を分割する
- 再実行に備えて、アプリケーションは **idempotent** に設計する
- クロール状態、再開点、重複防止に必要な状態は `GCS` / `Firestore` / `Cloud SQL` など外部ストレージで管理する

### 初期構成

- `Cloud Scheduler` から定期起動
- `Cloud Run Jobs` で crawler を実行
- コンテナイメージは `Artifact Registry` に保存
- ログとメトリクスは `Cloud Logging` / `Cloud Monitoring` で監視
- 収集データと状態は用途に応じて `GCS`, `Cloud SQL`, `Firestore` などへ保存

### 再検討条件

以下の条件に該当した場合は、Cloud Batch への移行または併用を再検討する。

- 現在のジョブ粒度では実行時間や並列数が不足する場合
- VM レベルのネットワーク制御やマシン選定が必要になった場合
- 大規模分散実行や複雑なジョブ依存関係が必要になった場合
- GPU 利用や特殊な計算ワークロードが発生した場合

## Consequences

### Positive

- **バッチ用途に適したマネージドサービスを使える**
  - 初期構築と運用を簡素化できる。
  - 個人開発でも扱いやすい構成になる。

- **コンテナ開発と実行基盤の接続が自然**
  - crawler をコンテナとして開発し、そのままジョブ実行基盤へ載せやすい。
  - Artifact Registry と Cloud Run を中心とした Terraform 構成にも自然に接続できる。

- **定期実行・手動実行・再実行の運用フローを統一しやすい**
  - Scheduler 起動、手動再実行、ログ確認、失敗時の再試行を比較的単純に扱える。

- **将来 shard 分割や task 並列化でスケールさせやすい**
  - task 数や timeout の調整で、対象増加や shard 分割に対応しやすい。
  - 実測ベースで必要十分な並列度を見極められる。

- **INFRA-ADR-001 / INFRA-ADR-002 と整合する**
  - App Project に stateless な compute を置く方針と一致する。
  - App Project 上の crawler job として位置づけやすい。

### Negative

- **Cloud Batch と比較すると制御自由度は低い**
  - VM レベルのマシンタイプ、ネットワーク、実行環境の細かい最適化には向かない。

- **アプリケーション設計の責任は残る**
  - checkpoint、再開性、レート制御、重複排除、shard 設計は crawler 側で適切に実装する必要がある。
  - 実行基盤だけではクロール品質や安定性は担保できない。

- **極端な大規模並列には限界がある**
  - 将来的に数百〜数千 task を高頻度で制御したい場合、Cloud Batch の方が適する可能性がある。

### Risks / Future Review

- クロール対象数の増加により、1 execution あたりの処理量が大きくなりすぎた場合は、task 分割戦略を見直す。
- クロール先の rate limit やブロック戦略に対して、アプリケーション側の制御が十分かを継続的に確認する。
- 再試行時の重複登録や途中失敗からの再開方式が曖昧な場合、運用コストが増えるため、状態管理設計を別 ADR または設計ドキュメントで明確化する。
- ワークロードが大規模分散処理に寄った場合は、Cloud Batch への移行を再検討する。

## Next Steps

1. `infra/terraform` に crawler 用の `Cloud Run Jobs` リソースを追加する
2. `Cloud Scheduler` から job を起動する構成を定義する
3. crawler 用 Service Account と必要最小限の IAM を設計する
4. `Artifact Registry` へのイメージ配布フローを CI/CD に組み込む
5. crawler 側で以下を実装・確認する
   - 冪等性
   - 再開可能性
   - shard 単位の実行
   - レート制御
   - 重複排除
6. 実運用後に以下を計測し、Cloud Batch 再検討の要否を判断する
   - 実行時間
   - 失敗率
   - 並列度
   - コスト
   - ネットワーク要件
